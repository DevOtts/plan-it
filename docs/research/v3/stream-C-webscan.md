## Findings (numbered, each with evidence: file:line or URL#anchor or quote)

1. **Hooks are the only deterministic enforcement layer Anthropic itself endorses; prompted rules are explicitly framed as unreliable.** "Hooks are user-defined shell commands that execute at specific points in Claude Code's lifecycle. They provide deterministic control over Claude Code's behavior, ensuring certain actions always happen rather than relying on the LLM to choose to run them." And, more pointedly: "the model can fail to follow a prompted rule... A real guardrail needs to be deterministic, and the enforcement methods are hooks and permissions." — https://code.claude.com/docs/en/hooks-guide, https://claude.com/blog/steering-claude-code-skills-hooks-rules-subagents-and-more. This directly validates plan-it v2.1's own architecture bet (PreToolUse hook + gate-check.mjs over prose instructions) — it is not a novel idea for v3, it's confirmation the existing bet is the state of the art.

2. **PreToolUse hooks fire before permission checks and can't be loosened by other hooks — only tightened.** "A `PreToolUse` hook can inspect a call and exit with code 2 to block it"; hooks fire "before any permission-mode check," so a deny decision blocks the tool "even in bypassPermissions mode"; but "a hook returning 'allow' doesn't bypass deny rules from settings — hooks can tighten restrictions but not loosen them." Caveat: when multiple PreToolUse hooks return `updatedInput`, "the last one to finish takes effect" and hook execution order is non-deterministic if unspecified — a real footgun for stacking multiple guard hooks. — https://code.claude.com/docs/en/hooks-guide#pretooluse

3. **Subagents don't automatically inherit the main agent's permission/hook context — a subagent needs its own guard.** "Subagents do not automatically inherit the main agent's permissions — so a PreToolUse hook may be the only thing gating a subagent's tool calls." This is relevant if v3 spins parallel squad subagents that each need to independently respect the CONTRACT-freeze gate, not just trust the parent's gate state. — https://code.claude.com/docs/en/hooks-guide (subagent hooks section)

4. **Anthropic's own multi-agent research system is a strong-orchestrator / cheap-worker pattern, with explicit effort-scaling rules baked into the prompt, not left to model judgement.** Architecture: "an orchestrator-worker pattern" where "a lead agent coordinates the process while delegating to specialized subagents." Critically, they found "agents struggle to judge appropriate effort for different tasks" and fixed it by hardcoding scaling rules: "simple fact-finding requires just 1 agent with 3-10 tool calls," "direct comparisons might need 2-4 subagents with 10-15 calls each," "complex research might use more than 10 subagents." Model tiering used in production: Opus 4 lead + Sonnet 4 subagents beat single-agent Opus 4 by 90.2% on their internal eval. Failure mode without explicit task boundaries: "one subagent explored the 2021 automotive chip crisis while 2 others duplicated work investigating current 2025 supply chains." — https://www.anthropic.com/engineering/multi-agent-research-system

5. **Model cascading/escalation is an active research area (2025-2026) with a consistent finding: cheap-first-then-escalate-on-low-confidence retains ~95-99% of top-model quality at a fraction of cost.** "Cluster, Route, Escalate" (arXiv 2606.27457) escalates only when a quality-estimation classifier judges output low-confidence, retaining "97-99% of the strongest model's accuracy." Practitioner framing distinguishes routing (one-shot, pre-execution classification) from cascading (sequential: try cheap, escalate on low confidence) — "hybrid approaches — route first... then cascade within a tier — often work best." This is the literal mechanism the mission brief calls "escalate-on-struggle," and it's already the pattern in your own CLAUDE.md tiering rule ("Escalate on struggle rather than pre-paying"). — arXiv 2606.27457, TianPan.co routing-vs-cascading blog

6. **Adaptive "effort" is replacing fixed thinking-token budgets as of Claude 4.6, and is specifically positioned as valuable for agentic plan/replan loops.** "Adaptive thinking lets Claude dynamically decide when and how much to think based on the complexity of each request... reliably drives better performance than extended thinking with a fixed budget_tokens." Accepted values: low/medium/high/max (max is Opus 4.6-only). "For agentic workflows where Claude must plan, call tools, observe results, and re-plan, extended thinking dramatically improves planning quality" — but "quality gains compress as the budget grows while costs scale linearly," so the right strategy is "the smallest budget that crosses the quality threshold." Fixed `budget_tokens` is deprecated on 4.6 and removed (400 error) on Opus 4.7+. — https://docs.claude.com/en/docs/build-with-claude/extended-thinking, https://platform.claude.com/docs/en/build-with-claude/adaptive-thinking

7. **Statecharts-as-runtime (not statecharts-as-diagram) is an explicit, named pattern for constraining LLM agent behavior, with an official Anthropic-adjacent implementation (Stately/XState).** `statelyai/agent` "create[s] state-machine-powered LLM agents using XState," where "the statechart constrains which events are valid in each state" via Zod-schema event definitions, and `agent.decide()` "plans actions only within those valid transitions." Framing: "the statechart isn't a diagram the agent might follow, it's the runtime that gates what the agent can do." This is precisely plan-it's own architecture (machine.json + gate-check.mjs) — external validation, not a new mechanism to adopt. — https://github.com/statelyai/agent, https://stately.ai/docs/agents, https://brightlume.ai/blog/why-ai-agent-needs-state-machine-not-prompt-chain

8. **Eval-driven, CI-gated prompt/agent testing (promptfoo) treats prompts like code: golden test sets, before/after diffing on PRs, and merge-blocking thresholds.** "Test-driven prompt engineering, rather than trial-and-error" — write test cases first, then optimize the prompt to pass them. The GitHub Action "can automatically run a before-vs-after evaluation of edited prompts on every pull request that modifies a prompt," posts pass/fail counts as a PR comment, and "supports repeating each test multiple times with a minimum-passes threshold to handle nondeterminism." A related pattern (Kinde) recommends "a golden dataset — starting with 10-20 high-priority examples... stored as CSV or JSON in your repo" as a merge-blocking gate. — https://github.com/promptfoo/promptfoo-action, https://www.promptfoo.dev/docs/integrations/github-action/

9. **Spec-Driven Development (SDD) tooling converged in 2025-2026 on the exact same shape plan-it already has — phased artifacts with hard gates between them — which is useful as an "are we behind or ahead of the field" check.** GitHub's `spec-kit`: phases `/speckit.constitution` → `/speckit.specify` → `/speckit.clarify` (**required before planning** — "run before creating a technical plan to reduce rework downstream") → `/speckit.plan` → `/speckit.tasks` → `/speckit.analyze` ("cross-artifact consistency & coverage analysis, run after tasks, before implement") → `/speckit.implement` (which itself "validates that all prerequisites are in place" before executing). It also ships `/speckit.checklist`, described as "like 'unit tests for English.'" AWS Kiro's spec flow (requirements.md/design.md/tasks.md, EARS syntax) similarly "checks your requirements for contradictions and gaps using automated reasoning techniques" before code generation, and runs independent tasks concurrently in "waves." Caveat from Martin Fowler's hands-on review: spec-kit's "many steps and markdown files felt like overkill for the size of the problem" and "the agent frequently didn't follow all the instructions" even with templates/checklists present — i.e., artifacts alone (without hook-level enforcement) still leak. This is a real skepticism data point: plan-it's edge over spec-kit/Kiro is precisely that its gates are hook-enforced (PreToolUse deny), not just markdown-prescribed. — https://github.com/github/spec-kit, https://github.github.com/spec-kit/, https://kiro.dev/docs/specs/, https://martinfowler.com/articles/exploring-gen-ai/sdd-3-tools.html

10. **Structured-output "contracts" (DSPy Signatures) and LangGraph's plan-review-execute interrupt pattern are two more mature, independent implementations of "freeze the plan, gate on human/machine approval before acting."** DSPy: "Signatures define the input/output contract for a language model call... signatures create a contract between your code and the LLM," with typed/Literal output fields enforced structurally rather than by instruction. LangGraph HITL: "the REVIEW node is a hard pause, not a suggestion — the graph freezes, state is checkpointed, and nothing moves until a human decides," plus "a circuit breaker stops execution... if 3 steps fail — no runaway loops." — https://dspy.ai/, https://docs.langchain.com/oss/python/langchain/human-in-the-loop, https://medium.com/@niteen.badgujar/plan-it-review-it-execute-it-building-a-human-in-the-loop-ai-agent-with-langgraph-c77aa41bcc74

## Mechanisms plan-it v3 could adopt (max 8)

1. **Adaptive effort parameter on discovery/spec states** → enforces proportional reasoning depth without a hardcoded token budget → phase: discovery + CONTRACT-drafting states (highest-ambiguity states in machine.json) → cost: near-zero engineering cost (it's an API param), but real $ cost scales with effort level; use "high" only on states with unresolved ambiguity, "medium" elsewhere. Source: finding 6.

2. **Explicit effort-scaling table embedded in SKILL.md/prompt (not left to model judgment)** → enforces right-sized subagent fan-out (N squads, M tool calls) per task-complexity tier, mirroring Anthropic's own fix for "agents struggle to judge appropriate effort" → phase: parallel-squad execution state (post-CONTRACT-freeze) → cost: prompt-engineering only, no new infra; directly prevents the "duplicate work / gaps" failure mode Anthropic documented. Source: finding 4.

3. **Escalate-on-low-confidence cascade for squad model tier, formalized as a rule not just a norm** → enforces that Haiku/Sonnet-tier squads escalate to Fable when a squad's own output fails gate-check or self-flags low confidence, rather than pre-assigning Fable everywhere → phase: squad-execution + gate-check states → cost: mostly free (re-run only on failure), matches your existing CLAUDE.md tiering rule — this is external validation to cite when writing the mechanism up, not a new build. Source: finding 5.

4. **Golden-test-set eval gate (promptfoo-style) on the CONTRACT/PRD artifacts themselves** → enforces that a fixed set of "must answer" scenarios (drawn from the 8 dogfood plan→execution pairs / EC-A1..EC-H*) are checked against every CONTRACT before freeze, with pass/fail surfaced like gate-check.mjs already does for structural checks → phase: G2 (decisions) gate, pre-freeze → cost: needs a small YAML/JSON golden set + a promptfoo (or hand-rolled) runner; moderate one-time build, near-zero marginal cost per run. Source: finding 8.

5. **Cross-artifact consistency pass before implementation, i.e. an "analyze" gate distinct from the freeze gate** → enforces that PRDs/epics/Test Contracts generated by parallel squads don't silently disagree (naming, scope boundaries, assumed APIs) before handoff → phase: new state between CONTRACT-freeze and packaging/handoff (analogous to spec-kit's `/speckit.analyze`, which plan-it doesn't currently have per SHARED-CONTEXT's "do NOT re-propose" list) → cost: moderate — likely a new gate-check.mjs subcommand that diffs artifacts for contradictions. Source: finding 9.

6. **Structured (schema-typed) hand-off objects between discovery and spec phases, DSPy-Signature style** → enforces that what discovery produces is exactly what spec-drafting consumes (typed fields, not prose the next phase has to re-parse and can misread) → phase: discovery → spec transition (the artifact interface, not the human gate) → cost: low; mostly a schema definition + validation step, reduces ambiguity-driven rework flagged in dogfood research. Source: finding 10.

7. **Hard circuit-breaker on the AMENDMENT self-loop** → enforces a bounded retry count (e.g., "stop and escalate to human after N failed AMENDMENT cycles") to prevent runaway replanning loops, mirroring LangGraph's "3 failed steps → jump to validation" pattern → phase: AMENDMENT self-loop (v1.0→v1.1 in the existing machine) → cost: very low, a counter + guard condition in machine.json. Source: finding 10.

8. **Explicit hook-ordering / single-writer discipline for any new PreToolUse guard added in v3** → enforces that if v3 adds more PreToolUse hooks (e.g., a new gate distinct from the existing CONTRACT-freeze one), they must not both attempt to rewrite the same tool input, since "the last one to finish takes effect" non-deterministically → phase: hook/infra layer, cross-cutting → cost: zero (a design constraint on the v2.1 hook, not new code) but important correctness note before adding hook #2. Source: finding 2.

## Out-of-scope but valuable (fable-it backlog candidates)

- **statelyai/agent (XState + Vercel AI SDK) as a reusable builder pattern** for any future Fable-authored agent that needs event-schema-constrained decisions, not just plan-it's own machine. Could inform a general "constrained-agent" skill in fable-it's builder toolkit. (Finding 7)
- **Subagent-level PreToolUse hook non-inheritance** is a general Claude Code gotcha (not plan-it-specific) worth documenting in fable-it's own skill-authoring guidance — any fable-it skill that spins subagents and assumes parent-level guards apply is silently unsafe. (Finding 3)
- **promptfoo-as-CI-gate for skill/prompt regressions** — broader than plan-it: fable-it itself could adopt a golden-test-set eval gate for its own skills (SKILL.md prompt changes triggering before/after eval on PRs), independent of the plan-it v3 scope. (Finding 8)
- **DSPy Signatures / ReAct** as a possible future builder-side technique for fable-it agents that need typed tool-call contracts rather than prose tool descriptions — a heavier lift, likely Python-only, not directly portable to the current TS/Node plan-it stack. (Finding 10)
- **Model-cascade classifier (fine-tuned confidence estimator, e.g. "Cluster, Route, Escalate")** as a more rigorous alternative to plan-it's judgment-based "escalate on struggle" — building an actual confidence classifier is a fable-it-scale infra investment, out of scope for a v3 planning-conductor change. (Finding 5)

## TL;DR

- Anthropic's own guidance ("Steering Claude Code") confirms plan-it v2.1's core bet — hooks/permissions are the only deterministic guardrail, prompted rules (CLAUDE.md/SKILL.md prose) are explicitly called unreliable under pressure/long sessions/prompt injection. Nothing to change here, just cite it.
- Statecharts-as-runtime (Stately/XState `statelyai/agent`) and Spec-Driven Development toolkits (GitHub spec-kit, AWS Kiro) independently converged on plan-it's own shape in 2025-2026 (phased artifacts + hard gates) — external validation, and spec-kit's `/speckit.analyze` cross-artifact consistency gate is one plan-it doesn't have yet.
- Martin Fowler's critical review of spec-kit is a useful caution: markdown-prescribed checklists still get skipped by the agent even with templates present — reinforcing that plan-it's hook-enforced gates (not prose gates) are the differentiator worth protecting in v3.
- Anthropic's multi-agent research system post is the best primary source for "explicit effort-scaling rules in the prompt" (don't leave fan-out sizing to model judgment) and validates a strong-orchestrator/cheap-worker tiering (Opus lead + Sonnet subagents, +90.2% vs single-agent).
- Model-cascade/escalation research (2025-2026 arXiv) formalizes "escalate on struggle" as a legitimate, actively-studied pattern (~95-99% quality retention vs top model) — matches the user's own CLAUDE.md tiering rule; no new build needed, just a citation.
- Claude 4.6's adaptive "effort" parameter is replacing fixed thinking-token budgets and is specifically recommended for plan/replan-heavy agentic loops — worth a light touch on discovery/CONTRACT states.
- promptfoo's CI-gate pattern (golden test set + before/after PR diffing + minimum-passes threshold for non-determinism) is a concrete, low-cost mechanism plan-it v3 could adopt for gating CONTRACT artifacts, distinct from anything currently in machine.json/gate-check.mjs.
- Several findings (XState agent framework, subagent hook non-inheritance, promptfoo-for-skills, DSPy Signatures, cascade classifiers) are stronger fits for fable-it's builder-side backlog than for plan-it v3's planning-conductor scope.
